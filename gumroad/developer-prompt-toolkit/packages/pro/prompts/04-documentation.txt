## DOC-01: JSDoc Generator
**Category:** Documentation
**Best for:** All
**When to use:** When you need to add JSDoc comments to JavaScript functions

### Prompt:
Add complete JSDoc comments to every function in this file. Include @param, @returns, @throws, @example, and @since tags where applicable.

```javascript
{{PASTE_CODE}}
```

Requirements:
1. Describe what the function does (not how it does it)
2. Document every parameter with type and description
3. Document the return value with type and description
4. Document any exceptions that may be thrown
5. Add at least one @example showing typical usage
6. Use TypeScript-style type annotations in @param tags
7. Mark optional parameters with square brackets

### Example usage:
Paste a utility module with 5-10 functions that are missing documentation.

### Pro tip:
Run your code through a JSDoc linter after generation to catch any formatting issues.

## DOC-04: Inline Code Commenter
**Category:** Documentation
**Best for:** All
**When to use:** When complex code needs explanatory comments

### Prompt:
Add clear, helpful inline comments to this code. Target audience: {{EXPERIENCE_LEVEL}} developers.

```{{LANGUAGE}}
{{PASTE_CODE}}
```

Rules:
1. Comment the WHY, not the WHAT (don't restate what the code does)
2. Explain business logic and domain-specific decisions
3. Comment complex algorithms step by step
4. Note any non-obvious side effects
5. Explain regex patterns in plain English
6. Mark TODO items for known improvements
7. Do NOT add comments to obvious code (variable assignments, simple loops)
8. Keep comments concise (1 line preferred, 2 max)

### Example usage:
Add comments to a payment processing function for a team of junior developers.

### Pro tip:
If you need more than 2 lines of comment, the code probably needs to be refactored instead.

## DOC-05: Changelog Generator
**Category:** Documentation
**Best for:** All
**When to use:** When writing release notes from git commits

### Prompt:
Generate a changelog entry from these git commits. Follow the Keep a Changelog format.

```
{{PASTE_GIT_LOG}}
```

Version: {{VERSION_NUMBER}}
Date: {{RELEASE_DATE}}

Categorize changes into:
- Added (new features)
- Changed (changes in existing functionality)
- Deprecated (soon-to-be removed features)
- Removed (removed features)
- Fixed (bug fixes)
- Security (vulnerability fixes)

Rules:
1. Write from the user's perspective (what they can now do)
2. Start each item with a verb in past tense
3. Group related commits into single entries
4. Skip internal refactoring unless it affects users
5. Link to relevant issues/PRs where applicable

### Example usage:
Paste the output of `git log --oneline v1.2.0..v1.3.0` to generate the v1.3.0 changelog.

### Pro tip:
Use conventional commits (feat:, fix:, docs:) to make changelog generation more accurate.

## DOC-06: Architecture Decision Record (ADR) Writer
**Category:** Documentation
**Best for:** Claude / ChatGPT
**When to use:** When documenting a significant technical decision

### Prompt:
Write an Architecture Decision Record (ADR) for this decision.

Decision: {{DESCRIBE_DECISION}}
Context: {{WHY_THIS_DECISION_WAS_NEEDED}}
Date: {{DATE}}
Status: {{PROPOSED/ACCEPTED/DEPRECATED}}

What we considered:
1. Option A: {{OPTION_A}}
2. Option B: {{OPTION_B}}
3. Option C: {{OPTION_C}}

What we chose: {{CHOSEN_OPTION}}
Why: {{REASONING}}

Follow the ADR format:
- Title (short descriptive title, numbered)
- Status
- Context (what is the issue?)
- Decision (what did we decide?)
- Consequences (what happens as a result?)

### Example usage:
Write an ADR for choosing PostgreSQL over MongoDB for a multi-tenant SaaS application.

### Pro tip:
Store ADRs in a `/docs/decisions/` directory and number them sequentially (0001, 0002, etc.).

## DOC-16: Database Schema Documenter
**Category:** Documentation
**Best for:** All
**When to use:** When documenting database tables, relationships, and constraints

### Prompt:
Create human-readable documentation for this database schema.

```sql
{{PASTE_CREATE_TABLE_STATEMENTS}}
```

For each table:
1. Table name and purpose
2. Column descriptions (name, type, constraints, description)
3. Primary keys and foreign keys
4. Indexes and why they exist
5. Relationships to other tables (with cardinality: one-to-many, many-to-many)
6. Important business rules encoded in constraints
7. Example row of data

Also create:
- An entity-relationship description (text-based, describable in Mermaid)
- A data dictionary (alphabetical list of all columns across all tables)

### Example usage:
Document a schema with users, orders, products, order_items, and reviews tables.

### Pro tip:
Include the expected row count and growth rate for each table to help with capacity planning.
