## DB-01: Query Optimizer
**Category:** Database & SQL
**Best for:** All
**When to use:** When a query is running slowly and needs optimization

### Prompt:
Optimize this slow SQL query. It currently takes {{CURRENT_TIME}} and should be under {{TARGET_TIME}}.

```sql
{{PASTE_QUERY}}
```

Table schemas:
```sql
{{PASTE_CREATE_TABLES}}
```

Current indexes:
```sql
{{PASTE_INDEXES}}
```

Table statistics:
- {{TABLE_1}}: {{ROW_COUNT}} rows
- {{TABLE_2}}: {{ROW_COUNT}} rows

EXPLAIN output (if available):
```
{{PASTE_EXPLAIN}}
```

1. Identify why this query is slow
2. Provide the optimized query
3. Suggest indexes to add (with CREATE INDEX statements)
4. Estimate the improvement
5. Are there any tradeoffs (more disk space, slower writes)?

### Example usage:
"Optimize a query joining users, orders, and products tables that takes 4.2 seconds on 2M rows. Target: under 200ms."

### Pro tip:
Always run EXPLAIN ANALYZE (PostgreSQL) or EXPLAIN (MySQL) and paste the output for the most accurate advice.

- Need soft delete: {{YES_NO}}
- Need audit trail: {{YES_NO}}
- Multi-tenant: {{YES_NO}}

## DB-03: Migration Writer
**Category:** Database & SQL
**Best for:** All
**When to use:** When you need to create a database migration

### Prompt:
Write a database migration for this change.

Database: {{DATABASE_TYPE}}
Migration tool: {{KNEX/PRISMA/SEQUELIZE/RAW_SQL}}

Current schema:
```sql
{{PASTE_CURRENT_SCHEMA}}
```

Desired change: {{DESCRIBE_CHANGE}}

Provide:
1. Up migration (apply the change)
2. Down migration (reverse the change)
3. Data migration (if existing data needs to be transformed)
4. Is this migration backward compatible? Can old code still work during deployment?
5. Estimated time to run on {{ROW_COUNT}} rows
6. Will it lock the table? For how long?

### Example usage:

### Pro tip:
For large tables, add columns as nullable first, backfill data, then add the NOT NULL constraint in a separate migration.

## DB-05: Index Strategy Advisor
**Category:** Database & SQL
**Best for:** All
**When to use:** When deciding what indexes to create for your tables

### Prompt:
Design an indexing strategy for this table.

Table schema:
```sql
{{PASTE_TABLE}}
```

Common query patterns:
1. {{QUERY_PATTERN_1}}
2. {{QUERY_PATTERN_2}}
3. {{QUERY_PATTERN_3}}
4. {{QUERY_PATTERN_4}}

Table statistics:
- Current rows: {{ROW_COUNT}}
- Growth rate: {{ROWS_PER_DAY}}
- Read/write ratio: {{RATIO}}

For each recommended index:
1. CREATE INDEX statement
2. Which query pattern it supports
3. Estimated size on disk
4. Impact on write performance
5. Is it a covering index?

Also identify:
- Indexes that should NOT be created (low selectivity, rarely used)
- Existing indexes that might be redundant
- Composite index column order considerations

### Example usage:
Design indexes for an orders table queried by: customer_id, status, date range, status+date, and full-text search on notes.

### Pro tip:
Order composite index columns by: equality filters first, then range filters, then sort columns.

**Category:** Database & SQL
**Best for:** All
**When to use:** When configuring database connection pooling

### Prompt:
Configure a database connection pool for my application.

Database: {{DATABASE_TYPE}}
Client library: {{LIBRARY}}

Expected concurrency: {{CONCURRENT_CONNECTIONS}}
Database max connections: {{MAX_CONNECTIONS}}

Current setup:
```{{LANGUAGE}}
{{PASTE_CURRENT_CONNECTION_CODE}}
```

Configure:
1. Pool size (min and max connections)
2. Connection timeout
3. Idle timeout
4. Connection validation (health check)
5. Retry logic for failed connections

## DB-08: Data Validation Constraints
**Category:** Database & SQL
**Best for:** All
**When to use:** When adding database-level data integrity constraints

### Prompt:
Add data integrity constraints to these tables.

```sql
{{PASTE_TABLES}}
```

Business rules:
{{LIST_BUSINESS_RULES}}

Add:
1. NOT NULL constraints where applicable
2. CHECK constraints for valid value ranges
3. UNIQUE constraints (single and composite)
4. Foreign key constraints with appropriate ON DELETE behavior
5. Default values
6. Custom validation triggers (if check constraint insufficient)

For each constraint:
- The ALTER TABLE or CREATE statement
- Which business rule it enforces
- What happens when a violation occurs
- Whether the application code also needs to validate this

### Example usage:
Add constraints for an e-commerce schema: order total must be positive, status must be one of specific values, email must be unique, order must belong to an existing user.

### Pro tip:
Always enforce constraints at the database level, not just in application code. Application code can be bypassed; database constraints cannot.

## DB-12: Soft Delete Implementation
**Category:** Database & SQL
**Best for:** All
**When to use:** When implementing soft delete (mark as deleted instead of removing)

### Prompt:
Implement soft delete for these tables.

```sql
{{PASTE_TABLES}}
```

Database: {{DATABASE_TYPE}}
ORM/Client: {{ORM}}

Implement:
1. Add deleted_at column (or is_deleted flag)
2. Create a view or default scope that excludes deleted records
3. Modify DELETE operations to set deleted_at instead
4. Update all SELECT queries to filter out deleted records
5. Create a "trash" query to list deleted records
6. Implement permanent delete (purge) function
7. Implement restore (undelete) function
8. Handle cascading soft delete (when parent is deleted)
9. Handle unique constraints with soft-deleted records
10. Add index for efficient filtering

### Example usage:
Implement soft delete for users and their associated orders and comments.

### Pro tip:
Use a partial unique index to enforce uniqueness only among non-deleted records: CREATE UNIQUE INDEX ON users (email) WHERE deleted_at IS NULL.

## DB-15: Database Backup and Recovery
**Category:** Database & SQL
**Best for:** Claude / ChatGPT
**When to use:** When setting up backup strategies and disaster recovery

### Prompt:
Design a backup and recovery strategy for my database.

Database: {{DATABASE_TYPE}}
Size: {{DATABASE_SIZE}}
RPO (Recovery Point Objective): {{MAX_DATA_LOSS_ACCEPTABLE}}
RTO (Recovery Time Objective): {{MAX_DOWNTIME_ACCEPTABLE}}
Environment: {{CLOUD_PROVIDER_OR_SELF_HOSTED}}

Provide:
1. Backup schedule (full and incremental)
2. Backup commands/scripts
3. Backup storage and retention policy
4. Point-in-time recovery setup
5. Recovery procedure (step by step)
6. Recovery testing procedure
7. Monitoring and alerting for backup failures
8. Documentation for the on-call team

### Example usage:

### Pro tip:
Test your recovery procedure regularly - a backup you've never tested is not a backup.
