## PERF-01: Frontend Performance Audit
**Category:** Performance Optimization
**Best for:** All
**When to use:** When a web page loads slowly or has poor Core Web Vitals

### Prompt:
Audit this page for frontend performance issues.

HTML:
```html
{{PASTE_HTML}}
```

CSS:
```css
{{PASTE_CSS}}
```

JavaScript:
```javascript
{{PASTE_JS}}
```

Lighthouse scores (if available):
- Performance: {{SCORE}}
- LCP: {{VALUE}}
- FID/INP: {{VALUE}}
- CLS: {{VALUE}}

Identify and fix:
1. Render-blocking resources
2. Unused CSS and JavaScript
3. Unoptimized images
4. Missing lazy loading
5. Layout shifts (CLS causes)
6. Large JavaScript bundles
7. Excessive DOM size
8. Inefficient CSS selectors
9. Missing compression (gzip/brotli)
10. Missing caching headers

For each issue, estimate the performance impact and provide the fix.

### Example usage:
Audit a product landing page with a Lighthouse performance score of 45. LCP is 4.5 seconds.

### Pro tip:
Focus on LCP first - it's the metric users feel most. Usually caused by large images, render-blocking CSS, or slow server response.

**Best for:** All
**When to use:** When your JavaScript bundle is too large

```
{{PASTE_BUNDLE_ANALYSIS}}
```

package.json dependencies:
```json
{{PASTE_DEPS}}
```

Reduce bundle size by:
1. Identify the largest dependencies
2. Find lighter alternatives for heavy libraries
3. Identify dead code that can be removed
4. Suggest code splitting points
5. Tree-shaking opportunities
6. Dynamic import candidates (lazy loading)
7. Move dependencies to CDN
8. Externalize large libraries

For each suggestion, estimate the size reduction.

### Example usage:
"My bundle is 1.2MB gzipped. Top dependencies: moment.js (300KB), lodash (70KB), chart.js (200KB)."

### Pro tip:
Replace moment.js with day.js (2KB) and lodash with individual imports (e.g., lodash-es/get instead of lodash).

**Best for:** All
**When to use:** When implementing caching to reduce load and latency

### Prompt:


- Write frequency: {{FREQ}}
- Staleness tolerance: {{SECONDS}}
- Data size per entry: {{SIZE}}

Implement:
1. Cache-aside (lazy loading) pattern
2. Cache key design (avoid collisions)
3. TTL settings with justification
4. Cache invalidation on data changes
5. Cache warming strategy

## PERF-07: Memory Optimization
**Category:** Performance Optimization
**Best for:** Claude / ChatGPT
**When to use:** When your application uses too much memory

### Prompt:
My {{LANGUAGE}} application is using {{CURRENT_MB}}MB of memory. Target: under {{TARGET_MB}}MB.

```{{LANGUAGE}}
{{PASTE_CODE}}
```

Usage pattern: {{DESCRIBE_USAGE}}

Optimize:
1. Find objects retained longer than needed
2. Replace in-memory data structures with more efficient alternatives
3. Implement streaming for large data processing
4. Use WeakRef/WeakMap for caches that should be GC-eligible
5. Fix event listener leaks
6. Reduce string duplication
7. Use buffers instead of strings for binary data
8. Implement pagination instead of loading all data

### Example usage:
"My Node.js worker processes 100K records. Memory grows to 2GB during processing. Target: under 512MB."

### Pro tip:
Process large datasets in streams or chunks. Never load an entire file or query result into memory if you can process it row by row.

## PERF-08: API Response Time Reducer
**Category:** Performance Optimization
**Best for:** All
**When to use:** When API responses need to be faster

### Prompt:
Reduce the response time of this API endpoint from {{CURRENT_MS}}ms to {{TARGET_MS}}ms.

```{{LANGUAGE}}
{{PASTE_ENDPOINT}}
```

Apply these techniques:
1. Parallelize independent async operations (Promise.all)
2. Reduce response payload (select only needed fields)
3. Add response compression
4. Implement conditional requests (ETag, If-Modified-Since)
5. Precompute expensive calculations
6. Use database views or materialized views for complex aggregations
7. Implement partial responses (fields parameter)
8. Move non-critical work to background jobs

### Example usage:
"Reduce /api/dashboard from 1.8 seconds to 300ms. It queries users, orders, products, and calculates stats."

### Pro tip:
If an endpoint calls 3 independent services sequentially (200ms + 300ms + 250ms = 750ms), parallelize them (max 300ms).

1. Connection pool size (based on server resources and concurrency)
2. Connection idle timeout
3. Connection validation (test before use)
4. Statement preparation and caching

## PERF-15: Rendering Performance
**Category:** Performance Optimization
**Best for:** All
**When to use:** When a page has jank or slow interactions

### Prompt:
Fix rendering performance issues on this page.

```html
{{PASTE_HTML}}
```

```css
{{PASTE_CSS}}
```

```javascript
{{PASTE_JS}}
```

Symptoms: {{DESCRIBE_JANK_OR_SLOW_INTERACTIONS}}

Fix:
1. Avoid forced synchronous layouts (read then write DOM)
2. Batch DOM reads and writes
3. Use requestAnimationFrame for visual updates
4. Virtualize long lists (only render visible items)
5. Debounce scroll and resize event handlers
6. Use CSS transforms for animations (avoid top/left)
7. Reduce DOM size (fewer elements)
8. Use will-change for elements that will animate
9. Avoid layout recalculations from JavaScript
10. Use Intersection Observer instead of scroll position checks

### Example usage:
"A dashboard with 500 data cards and infinite scroll. Scrolling is janky and adding new cards freezes the UI for 200ms."

### Pro tip:
Use Chrome DevTools Performance panel to record scrolling and identify which functions cause long frames (>16ms).
