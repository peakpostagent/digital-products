## REVIEW-01: Security-Focused Code Review
**Category:** Code Review
**Best for:** All
**When to use:** Before deploying code that handles user input, authentication, or sensitive data

### Prompt:
Perform a security-focused code review on this code. Assume an attacker will try to exploit it.

```{{LANGUAGE}}
{{PASTE_CODE}}
```

Check for:
1. SQL injection vulnerabilities
2. XSS (cross-site scripting) vulnerabilities
3. CSRF vulnerabilities
4. Insecure direct object references
5. Hardcoded secrets or credentials
6. Improper input validation or sanitization
7. Insecure cryptography usage
8. Path traversal vulnerabilities
9. Information leakage in error messages
10. Missing rate limiting considerations

For each finding, rate severity (Critical/High/Medium/Low) and provide the secure fix.

### Example usage:
Paste a login route handler or a file upload endpoint.

### Pro tip:
Specify your deployment environment (AWS, Vercel, etc.) since some platforms provide built-in protections.

## REVIEW-02: Performance Code Review
**Category:** Code Review
**Best for:** Claude / ChatGPT
**When to use:** When reviewing code that processes large datasets or handles high traffic

### Prompt:
Review this code for performance issues. It processes approximately {{VOLUME}} requests/records per {{TIME_PERIOD}}.

```{{LANGUAGE}}
{{PASTE_CODE}}
```

Analyze:
1. Time complexity of algorithms used
2. Unnecessary memory allocations
3. N+1 query problems
4. Missing caching opportunities
5. Synchronous operations that should be async
6. Redundant computations that could be memoized
7. Inefficient data structures

### Example usage:

### Pro tip:

**Best for:** Claude / ChatGPT

### Prompt:
Review these tests for quality and completeness.

Code under test:
```{{LANGUAGE}}
{{PASTE_SOURCE_CODE}}
```

```{{LANGUAGE}}
{{PASTE_TEST_CODE}}
```

Evaluate:
1. Are all public functions/methods tested?
2. Are edge cases covered (null, empty, boundary values)?
3. Are error cases tested?
4. Are tests independent (no shared state)?
5. Do test names clearly describe what they test?

## REVIEW-11: Code Smell Detector
**Category:** Code Review
**Best for:** All
**When to use:** When you want a general quality check for anti-patterns

### Prompt:
Scan this code for code smells and anti-patterns.

```{{LANGUAGE}}
{{PASTE_CODE}}
```

Look for these smells:
1. Long functions (>30 lines)
2. Deep nesting (>3 levels)
3. Too many parameters (>4)
4. Feature envy (function uses another module's data excessively)
5. God objects/modules (doing too much)
6. Primitive obsession (using primitives instead of small objects)
7. Shotgun surgery risk (change here requires changes in many other places)
8. Dead code
9. Speculative generality (over-engineering for future needs)

For each smell found, explain why it's problematic and show the refactored version.

### Example usage:
Paste any module or class that feels "messy" but you cannot pinpoint exactly why.

### Pro tip:
Prioritize fixing smells that make the code hard to test - those are usually the most impactful.
