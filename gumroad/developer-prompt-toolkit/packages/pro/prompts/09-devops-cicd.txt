## DEVOPS-01: Dockerfile Builder
**Category:** DevOps & CI/CD
**Best for:** All
**When to use:** When creating or optimizing a Dockerfile

### Prompt:
Create a production-ready Dockerfile for my application.

Application type: {{APP_TYPE}}
Language/Runtime: {{LANGUAGE_AND_VERSION}}
Build command: {{BUILD_COMMAND}}
Start command: {{START_COMMAND}}
Port: {{PORT}}
Dependencies: {{LIST_SYSTEM_DEPS}}

Requirements:
1. Multi-stage build (separate build and runtime stages)
2. Minimal final image size
3. Non-root user for security
4. Proper .dockerignore file
5. Layer caching optimization (dependencies before code)
6. Health check instruction
7. Environment variable handling
8. Signal handling for graceful shutdown
9. No development dependencies in final image
10. Security scanning considerations

Also provide the .dockerignore file.

### Example usage:
Create a Dockerfile for a Node.js 20 Express API that builds with npm and runs on port 3000.

### Pro tip:
Use `node:20-alpine` instead of `node:20` to reduce image size from 1GB+ to ~150MB.

## DEVOPS-02: GitHub Actions Workflow Builder
**Category:** DevOps & CI/CD
**Best for:** All
**When to use:** When setting up CI/CD with GitHub Actions

### Prompt:
Create a GitHub Actions workflow for my project.

Project type: {{PROJECT_TYPE}}
Language: {{LANGUAGE}}
Package manager: {{NPM/YARN/PNPM}}
Node version: {{VERSION}}

Workflow should:
1. Trigger on: {{PUSH_TO_MAIN/PR/MANUAL}}
2. Run linting
3. Run tests with coverage report
4. Build the project
5. Deploy to {{DEPLOYMENT_TARGET}} (if applicable)
6. Cache dependencies between runs
7. Run on {{UBUNTU/MACOS/WINDOWS}}
8. Set up environment variables and secrets
9. Post status to PR (if PR trigger)
10. Fail fast if linting fails (don't run tests)

Provide the complete `.github/workflows/ci.yml` file.

### Example usage:
Create a CI workflow for a Node.js project: lint with ESLint, test with Vitest, build, deploy to Vercel on main branch push.

### Pro tip:
Use `actions/cache` for node_modules to speed up workflows by 30-60 seconds.

## DEVOPS-03: Docker Compose Setup
**Category:** DevOps & CI/CD
**Best for:** All
**When to use:** When setting up a multi-service development environment

### Prompt:
Create a docker-compose.yml for my development environment.

Services needed:
1. {{SERVICE_1}}: {{DETAILS}}
2. {{SERVICE_2}}: {{DETAILS}}
3. {{SERVICE_3}}: {{DETAILS}}

Requirements:
- Persistent volumes for data
- Health checks for dependencies
- Proper startup order (depends_on with condition)
- Environment variables from .env file
- Port mapping for development access
- Hot reload for application code (volume mounts)
- Network isolation between services
- Resource limits (memory, CPU)

Also provide:
- .env.example file
- Commands cheat sheet (up, down, logs, rebuild)

### Example usage:
Create docker-compose for: Node.js API (with hot reload), PostgreSQL 15, Redis 7, and pgAdmin for database management.

### Pro tip:
Use `depends_on` with `condition: service_healthy` instead of just `depends_on` to ensure databases are actually ready.

2. Error rates (4xx, 5xx)
3. Resource usage (CPU, memory, disk)
4. Database performance (query time, connection pool)
5. External dependency health
6. Business metrics (signups, orders, etc.)

- Warning: {{DEFINE}}

### Example usage:
Set up monitoring for a Node.js API on AWS with PostgreSQL. Alert on Slack for warnings and PagerDuty for critical.

## DEVOPS-07: CI Pipeline Optimizer
**Category:** DevOps & CI/CD
**Best for:** All
**When to use:** When your CI pipeline is too slow

### Prompt:
My CI pipeline takes {{CURRENT_TIME}} and I need it under {{TARGET_TIME}}.

Current workflow:
```yaml
{{PASTE_CI_CONFIG}}
```

Pipeline timing breakdown:
{{LIST_STEPS_WITH_TIMES}}

Optimize by:
1. Identify the bottleneck steps
2. Cache dependencies and build artifacts
3. Parallelize independent steps
4. Skip unnecessary steps (conditional execution)
5. Use smaller/faster Docker images
6. Optimize test execution (parallel tests, selective testing)
7. Use incremental builds
8. Optimize Docker layer caching

Provide the optimized workflow file with comments explaining each improvement.

### Example usage:
Optimize a GitHub Actions workflow that takes 12 minutes: install (3 min), lint (1 min), test (5 min), build (2 min), deploy (1 min).

### Pro tip:
Run lint and tests in parallel since they don't depend on each other. Cache node_modules between runs.

## DEVOPS-14: Zero-Downtime Deployment
**Category:** DevOps & CI/CD
**Best for:** Claude / ChatGPT
**When to use:** When implementing deployment strategies that avoid downtime

### Prompt:
Implement zero-downtime deployment for my application.

Current setup:
{{DESCRIBE_CURRENT_DEPLOYMENT}}

Application: {{APP_TYPE}}
Infrastructure: {{INFRA}}
Database migrations: {{YES_NO}}

Strategy: {{BLUE_GREEN/ROLLING/CANARY}}

Implement:
1. Deployment script/pipeline
2. Load balancer configuration for traffic switching
3. Health check verification before switching traffic
4. Database migration strategy (backward compatible)
5. Rollback procedure if deployment fails
6. Smoke tests after deployment
7. Gradual traffic shifting (canary)

### Example usage:
Implement blue-green deployment for a Node.js API behind an AWS ALB with PostgreSQL database migrations.

### Pro tip:
Make every database migration backward compatible. Deploy the migration first, then the code, so you can rollback the code without rolling back the database.

**Category:** DevOps & CI/CD
**Best for:** All
**When to use:** When securing a production deployment

### Prompt:
Create a security hardening checklist for my production deployment.

Stack: {{STACK}}
Infrastructure: {{INFRA}}
Compliance requirements: {{NONE/SOC2/HIPAA/PCI}}

Check and configure:
1. OS-level security (updates, firewall, SSH hardening)
2. Application security headers
3. HTTPS enforcement
4. Secret management (no secrets in code/env files)
5. Database security (network access, credentials rotation)
6. Container security (non-root, read-only filesystem)
7. Dependency vulnerability scanning
8. Log security (no sensitive data in logs)
9. Access control and least privilege
10. Backup encryption
11. Network security (VPC, security groups, WAF)
12. Monitoring and incident response

### Example usage:
